# MY-RULES.md  
Author: Brother Bill  
Purpose: Operator‑grade rules of engagement for Copilot when assisting with this repo.

---

# 1. General Doctrine
- All workflows must be **deterministic**, **repeatable**, and **operator‑grade**.
- No ambiguity, no “choices,” no “options,” no “preferences.”
- Instructions must be **explicit**, **step‑by‑step**, and **unambiguous**.
- Operators follow the doctrine exactly; improvisation is not allowed.
- All content must match the tone: direct, clean, technical, and free of fluff.

---

# 2. Linux Rules
- Linux instructions must use **Pop!\_OS GUI only** unless explicitly stated otherwise.
- Never assume terminal usage unless Brother Bill explicitly requests it.
- Never assume Dropbox is integrated into the Linux filesystem.
- All Dropbox interactions on Linux must be done **in the browser**.
- Linux recording machine must remain **stateless** after each session.
- Linux uses **OBS Studio**, not Camtasia Recorder.
- Linux produces **.mkv**, not `.trec`.
- **OBS must use a single, validated PipeWire capture source**  
  (No multiple sources, no ambiguous sources, no “Entire Workspace,” no “Virtual Output.”  
  Only the explicit monitor the operator is using.)

---

# 3. Windows Rules
- Never assume Dropbox is integrated into the Windows filesystem.
- All Dropbox interactions on Windows must be done **in the browser**.
- Windows dehydration workflow must always be **inlined** when referenced.
- Windows working folder is always:
  ```
  C:\dev\catools-lc\
  ```
- Windows machine must remain **stateless** after each session.

---

# 4. Dropbox Rules
- Dropbox is the **only** transfer method between Linux, Windows, and macOS.
- Dropbox must be accessed **via browser only**.
- Required folder structure:
  ```
  /caTools/
  /caTools/LearningCurve/
  ```
- If folders do not exist, instructions must include steps to create them.

---

# 5. File & Folder Rules
- All filenames must be deterministic and standardized.
- Recording filenames must follow:
  ```
  lesson_01.mkv
  lesson_01.wav
  lesson_01.mp4
  ```
- Linux staging folder:
  ```
  ~/junk/
  ```
- Linux project folder:
  ```
  ~/dev/catools-lc/
  ```
- Windows project folder:
  ```
  C:\dev\catools-lc\
  ```

---

# 6. Sanity Check Rules
- VLC is the **only** approved playback tool on Linux and Windows.
- COSMIC Media Player must never be used.
- All recordings must undergo a 5–10 second playback sanity check.
- If audio is missing, operator must stop and re‑record.
- **All `.mkv` sanity checks must be performed explicitly in VLC**  
  (Right‑click → Open With → VLC Media Player.  
  Default system players are not trusted and must not be used.)

---

# 7. Writing Rules
- All documentation must be Markdown.
- Tone must match Brother Bill’s doctrine: direct, deterministic, operator‑grade.
- No fluff, no filler, no “helpful suggestions” outside doctrine.
- No offering choices unless explicitly requested.
- No assumptions about user intent beyond the doctrine.
- **All ordered list items in Markdown must use `1.` for every item**  
  (Markdown auto‑numbers; this prevents renumbering errors and supports deterministic editing.)
- **Every Markdown file must begin with exactly one header line containing the full path and filename.**  
  (No second line repeating the folder name. No extraneous structure.)
- **All Markdown files generated by Copilot must be delivered inside a 4‑backtick copy block.**  
  (This preserves formatting, prevents corruption when pasting into VS Code, and ensures deterministic reproduction of folder structures, code blocks, and ordered lists.)
- **The entire Markdown file must be inside the copy block — never partial fragments.**  
  (This guarantees atomic delivery, prevents drift, and ensures operators paste the file exactly as intended.)
- **No Markdown file may contain workflows for more than one operating system or more than one physical machine**  
  (Each MD file must be OS‑pure and machine‑pure. Linux workflows and Windows workflows must always be split into separate files. This prevents operator confusion, eliminates cross‑OS drift, and ensures deterministic execution.)
- **If a workflow depends on a specific file format (e.g., `.mkv`), the MD file must include explicit commentary inside the content explaining the current supported format and how to update it in the future.**  
  (The filename must never include the format.)

---

# 8. Copilot Behavior Rules
- When this file is provided at the start of a session, Copilot must follow it **exactly**.
- If the session becomes long and context drifts, Brother Bill may re‑anchor by pasting this file again.
- Copilot must enforce these rules and correct any drift in outputs.
- Copilot must warn if any requested action contradicts this doctrine.

---

# 9. macOS Rules
- macOS instructions must use **GUI only** unless explicitly stated otherwise.
- Never assume Dropbox is integrated into the macOS filesystem (no Finder‑mounted Dropbox).
- All Dropbox interactions on macOS must be done **in the browser**.
- macOS will be used for Dropbox transfers and future workflows defined later.
- macOS machine must remain stateless after each session.
